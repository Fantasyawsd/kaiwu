## 先看这几个结论

- `Val` 比 `Train` 更值得信。
  `train_*` 会被课程学习随机采样直接影响，难度会主动变化；`val_*` 更适合看真实进步。
- 先看生存，再看得分。
  推荐顺序是：
  `val_completed_rate` -> `val_post_speedup_terminated` -> `val_post_speedup_steps` -> `val_total_score / val_treasure_score`
- `reward` 不等于 `score`。
  `reward` 是训练目标，包含 shaping 和终局奖励；`score` 是环境得分。
- `*_penalty` 一般越接近 `0` 越好。
  它们大多是负数，越负说明问题越重。
- `Progress` 组指标现在做了限流。
  局内不是每一步都上报了，而是开局、结算、以及每 `50` 步上报一次。

## 命名规则

- `train_*`
  训练局指标。单局上报，面板看到的是时间窗口内平均值。
- `val_*`
  验证指标。当前实现是每轮验证把多局均值一次性上报。
- `*_rate`
  通常在 `0~1` 之间，越接近 1 代表事件越常发生。
- `*_gain`
  某一阶段内环境分数的累计增量。
- `*_reward` / `*_bonus`
  奖励项，通常越大越好。
- `*_penalty`
  惩罚项，通常越接近 `0` 越好。

## UI 常见别名

- `explainedCritic` 或 `ExplainedCritic`
  对应 `explained_var`
- `advMean`
  对应 `adv_mean`
- `retMean`
  对应 `ret_mean`
- `EpisodeID`
  对应 `current_episode_id`
- `CompletedEP`
  对应 `completed_episode_count`

## 推荐看盘顺序

1. 先确认系统在正常跑：
   `current_episode_id`、`completed_episode_count`、`current_episode_step`
2. 再看验证生存：
   `val_completed_rate`、`val_terminated_rate`、`val_post_speedup_terminated`、`val_post_speedup_steps`
3. 再看验证收益：
   `val_total_score`、`val_treasure_score`、`val_treasures_collected`
4. 再看训练稳定性：
   `explained_var`、`value_loss`、`clip_frac`、`grad_clip_norm`
5. 最后看行为问题：
   `flash_waste_penalty`、`hit_wall_penalty`、`stagnation_penalty`、`oscillation_penalty`、`treasure_miss_penalty`

## 1. Progress 组

| 指标 | 定义 | 怎么看是好还是坏 |
| --- | --- | --- |
| `current_episode_id` | 当前 worker 正在跑到的 episode 编号 | 主要看它是否持续增加。注意它是多 worker 平均值，不是严格的全局唯一 episode。 |
| `completed_episode_count` | 当前 worker 已完成 episode 数 | 主要看它是否持续增加。如果长时间不动，说明卡住或环境异常。 |
| `current_episode_step` | 当前正在跑的局已经走了多少步 | 正常会在局内增长，结算后回落。现在默认每 `50` 步上报一次。 |
| `current_episode_is_eval` | 当前是不是在 eval，训练为 `0`，验证为 `1` | 用来判断当前窗口的数据是不是掺了验证局。 |
| `train_episode_total` | 当前训练总局数计数 | 用来判断当前已经进入哪个 curriculum 阶段。 |

注意：
- 这组指标是运行状态指标，不是性能指标。
- 因为监控表达式是 `avg(...)`，如果有多个 aisrv/worker，它们显示的是平均状态，不是单个进程的精确值。

## 2. Train / Val 通用局级指标

下面这组定义对 `train_*` 和 `val_*` 都成立。区别只是：
- `train_*` 受课程学习分布影响更大
- `val_*` 更适合判断真实能力

| 指标 | 定义 | 怎么看是好还是坏 |
| --- | --- | --- |
| `{p}_reward` | 整局总训练奖励，等于 `pre_shaped + post_shaped + pre_terminal + post_terminal` | 一般越高越好，但只能和同分布比较。优先看 `val_reward`。 |
| `{p}_total_score` | 终局环境总分 | 很重要，通常越高越好。 |
| `{p}_step_score` | 终局生存分 | 越高通常表示活得更久。 |
| `{p}_treasure_score` | 终局宝箱分 | 越高通常表示拿箱子更好。 |
| `{p}_treasures_collected` | 终局实际拿到的宝箱数 | 越高通常越好，但要结合生存指标一起看。 |
| `{p}_episode_steps` | 这局结束时的步数 | 越高通常越好，接近 `max_step` 说明更常活满。 |
| `{p}_speedup_reached` | 这局是否活到怪物加速阶段，单局是 `0/1` | 越高越好。 |
| `{p}_phase_time_to_speedup` | episode 内每步 `time_to_speedup_norm` 的平均值 | 这是阶段位置指标，不是“真实剩余多少步”。不要单独拿它判断好坏。 |
| `{p}_pre_speedup_steps` | 加速前阶段累计步数 | 高不一定是好，可能只是 speedup 本来就来得晚。 |
| `{p}_post_speedup_steps` | 加速后阶段累计步数 | 很关键，越高越好，说明 speedup 后也能活。 |
| `{p}_pre_speedup_reward` | 加速前总奖励，包含终局奖励 | 高说明前期推进不错。 |
| `{p}_post_speedup_reward` | 加速后总奖励，包含终局奖励 | 很关键，越高越好。 |
| `{p}_pre_speedup_shaped_reward` | 加速前 shaping 累计，不含终局奖励 | 看前期奖励引导是否学到。 |
| `{p}_post_speedup_shaped_reward` | 加速后 shaping 累计，不含终局奖励 | 看后期生存引导是否学到。 |
| `{p}_pre_speedup_step_score_gain` | 加速前 `step_score` 累积增量 | 越高越好，说明前期存活推进稳定。 |
| `{p}_post_speedup_step_score_gain` | 加速后 `step_score` 累积增量 | 很关键，越高越好。 |
| `{p}_pre_speedup_treasure_gain` | 加速前 `treasure_score` 累积增量 | 越高越好，说明前期拿箱更有效。 |
| `{p}_post_speedup_treasure_gain` | 加速后 `treasure_score` 累积增量 | 高是加分项，但要结合生存看，别为了贪箱而暴毙。 |
| `{p}_pre_speedup_total_score_gain` | 加速前 `total_score` 累积增量 | 越高越好。 |
| `{p}_post_speedup_total_score_gain` | 加速后 `total_score` 累积增量 | 很关键，说明不是只会苟活。 |

这里的 `{p}` 可以替换成 `train` 或 `val`。

## 3. 行为 shaping 指标

这组主要用来看策略是不是在犯你最关心的几类错误。

| 指标 | 定义 | 怎么看是好还是坏 |
| --- | --- | --- |
| `{p}_pre_speedup_buffer_reward` | speedup 前缓冲窗口内，提前拉开和怪物距离得到的奖励 | 更正、更高通常更好。若它明显升高但总分下降，可能过于保守。 |
| `{p}_second_monster_pressure_penalty` | 第二只怪压得太近时的惩罚 | 越接近 `0` 越好；很负说明经常被第二只怪压住。 |
| `{p}_flash_direction_reward` | 危险时朝更合理方向交闪得到的奖励 | 越高通常越好，说明闪现方向更合理。 |
| `{p}_flash_waste_penalty` | 闪现后既没明显脱险也没拿到资源的惩罚 | 越接近 `0` 越好；很负说明乱交闪。 |
| `{p}_hit_wall_penalty` | 撞墙或无效移动惩罚 | 越接近 `0` 越好；很负说明撞墙多。 |
| `{p}_stagnation_penalty` | 原地磨蹭、位移太小的惩罚 | 越接近 `0` 越好；很负说明不动或小范围抖动严重。 |
| `{p}_oscillation_penalty` | 来回反向抖动的惩罚 | 越接近 `0` 越好；很负说明“左右横跳”严重。 |
| `{p}_treasure_miss_penalty` | 靠近宝箱又明显走开时的惩罚 | 越接近 `0` 越好；很负说明路过不捡宝箱。 |
| `{p}_no_vision_patrol_bonus` | 看不到怪、附近没目标时持续移动的奖励 | 更高通常更好，说明无视野时不会一直原地乱晃。 |

实战解释：
- `flash_waste_penalty` 很负 + `flash_count` 很高：
  典型的“闪现随便交”
- `hit_wall_penalty`、`stagnation_penalty`、`oscillation_penalty` 都很负：
  典型的“撞墙后反复来回走”
- `treasure_miss_penalty` 很负：
  典型的“路过宝箱但没捡”

## 4. 只在验证里特别重要的终局指标

| 指标 | 定义 | 怎么看是好还是坏 |
| --- | --- | --- |
| `val_pre_speedup_terminal_bonus` | 如果终局发生在 pre 阶段，这部分终局奖励记到 pre | 参考意义有，但不如后面几个直接。 |
| `val_post_speedup_terminal_bonus` | 如果终局发生在 post 阶段，这部分终局奖励记到 post | 配合 `val_post_speedup_reward` 看。 |
| `val_post_speedup_terminated` | 如果在 speedup 后死亡，则记 `1` | 越低越好，这是最该盯的坏信号之一。 |
| `val_terminated_rate` | 验证局死亡率 | 越低越好。 |
| `val_completed_rate` | 验证局活满 `max_step` 的比例 | 越高越好，这是第一核心指标。 |
| `val_abnormal_truncated_rate` | 非死亡、也没活满、但异常截断的比例 | 越接近 `0` 越好；高了先怀疑环境或流程问题。 |
| `val_danger_level` | 终局危险度，定义为 `clip(1 - 最近怪距离 / MAX_DISTANCE, 0, 1)` | 越低越好，说明结局时没被怪贴得太近。 |
| `val_nearest_treasure_dist` | 终局时最近宝箱距离；如果没有可用宝箱则为 `-1` | 小通常更好；`-1` 不一定坏，可能表示宝箱已清空。要结合 `val_treasures_collected` 看。 |

最常用的验证判断：
- `val_completed_rate` 上升
- `val_post_speedup_terminated` 下降
- `val_post_speedup_steps` 上升

如果这三项同时向好，通常说明策略真的在变强。

## 5. Env 原始环境指标

这组更多是用来检查：
- 课程学习分布是否生效
- 环境当前状态是不是符合预期

它们不是最直接的优化目标。

| 指标 | 定义 | 怎么看是好还是坏 |
| --- | --- | --- |
| `total_score` | 环境总分 | 主要做 sanity check，和 `train_/val_` 同名指标不是同一条上报链。 |
| `treasure_score` | 环境宝箱分 | 主要看趋势和分布。 |
| `step_score` | 环境生存分 | 趋势上越高通常越好。 |
| `total_map` | 当前地图编号的平均值 | 不是越大越好，只是确认地图分布是否正常。 |
| `map_random` | 是否随机地图 | 训练应接近 `1`。 |
| `max_steps` | 当前局最大步数 | 现在应该看到 `2000`。如果不是，说明旧配置还在跑。 |
| `finished_steps` | 这局实际结束步数 | 越高通常越好。 |
| `total_treasure` | 当前局总宝箱数 | 主要用来确认 curriculum 是否生效。 |
| `treasures_collected` | 当前局实际拿到的宝箱数 | 越高通常越好。 |
| `flash_count` | 当前局闪现使用次数 | 没有绝对最优值，要和 `flash_waste_penalty` 联合看。 |
| `flash_cooldown` | 闪现冷却配置 | 主要看环境是否按预期。 |
| `total_buff` | 当前局 buff 总数 | 用来检查 curriculum。 |
| `collected_buff` | 当前局已拿到的 buff 数 | 趋势上高一些通常更好。 |
| `buff_refresh_time` | buff 刷新时间配置 | 主要用于配置 sanity check。 |
| `monster_speed` | 当前怪物速度 | speedup 后应升高。 |
| `monster_interval` | 第二只怪出现时间配置 | 用来确认当前课程采样是否正确。 |

## 6. Learner 损失类指标

这组是训练稳定性指标，不是环境表现指标。

| 指标 | 定义 | 怎么看是好还是坏 |
| --- | --- | --- |
| `reward` | 当前训练 batch 的样本奖励均值，不是整局 reward | 只能看趋势，不能和 `train_reward` 混用。 |
| `total_loss` | `vf_coef * value_loss + policy_loss - beta * entropy_loss` | 绝对值意义不大，主要看是否突然爆炸或剧烈振荡。 |
| `value_loss` | PPO 的 value loss | 长期偏高通常不好，说明 critic 跟不上。 |
| `policy_loss` | PPO 的 clipped surrogate loss | 绝对值别过度解读；突跳大多说明策略更新过猛或数据分布变了。 |
| `entropy_loss` | 实际上是策略熵均值，不是带负号的“损失” | 太高表示太随机，太低表示太确定。前期过低容易早熟，后期过高会学不稳。 |
| `grad_clip_norm` | 裁剪前梯度范数 | 持续远高于裁剪阈值 `0.5` 通常不好，说明更新太猛。 |
| `clip_frac` | `|ratio - 1| > clip_param` 的样本比例 | 太低可能更新太弱，太高说明更新太猛。中等更健康。 |
| `explained_var` | `1 - Var(return - value_pred) / Var(return)` | 越接近 `1` 越好；接近 `0` 说明 critic 几乎没解释力；负数通常很差。 |
| `adv_mean` | 原始 advantage 的 batch 均值，归一化前 | 一般希望相对稳定，长期接近 `0` 更自然；长期明显偏负说明 critic 常高估，明显偏正说明 critic 常低估。 |
| `ret_mean` | batch 的 `reward_sum` 均值，也就是 return target 均值 | 在同一分布下越高通常越好；跨 curriculum 阶段不要直接比较。 |

### 怎样一起看损失类指标

- `explained_var` 低或为负
- `value_loss` 高
- `clip_frac` 乱跳
- `grad_clip_norm` 很大

如果这 4 个同时不好，先别太信 reward 曲线，训练本身可能不稳定。

## 9. 常见组合诊断

### 1. 只会苟活，不会拿分

表现：
- `val_post_speedup_steps` 上升
- `val_completed_rate` 上升
- 但 `val_treasure_score`、`val_treasures_collected` 不升反降

说明：
- 学会了活，但不愿意承担拿箱风险

### 2. 前期会拿箱，后期一碰 speedup 就死

表现：
- `val_pre_speedup_treasure_gain` 高
- `val_post_speedup_steps` 低
- `val_post_speedup_terminated` 高

说明：
- 前期策略有用，后期 survival 不够

### 3. 闪现乱交

表现：
- `flash_count` 高
- `flash_waste_penalty` 很负
- `flash_direction_reward` 不高

说明：
- 闪现用得多，但没用在刀刃上

### 4. 撞墙抖动

表现：
- `hit_wall_penalty` 很负
- `stagnation_penalty` 很负
- `oscillation_penalty` 很负

说明：
- 行为上还在“卡住后左右反复试”

### 5. 路过箱子不捡

表现：
- `treasure_miss_penalty` 很负
- `val_treasure_score` 上不去

说明：
- 已经识别到宝箱了，但临门一脚没完成

### 6. critic 不工作

表现：
- `explained_var` 接近 `0` 或负数
- `value_loss` 高
- `adv_mean` 长期明显偏正或偏负

说明：
- value 估计明显不准，训练稳定性会受影响

## 10. 当前监控补充说明

截至 `2026-04-12` 当前代码，下面这些 `val_*` 行为指标已经和 `train_*` 一样接入验证汇总：

- `val_flash_direction_reward`
- `val_hit_wall_penalty`
- `val_stagnation_penalty`
- `val_oscillation_penalty`
- `val_treasure_miss_penalty`
- `val_no_vision_patrol_bonus`

如果这些 `Val` 面板仍然显示“无数据”，优先排查这几类情况：

- 当前训练进程还没重启，监控仍在跑旧代码。
- 还没走到一轮完整 eval 汇总。
  当前实现是攒满一批验证局后才一次性上报，不是每局实时更新。
- monitor 后端有窗口延迟。
  新值上来后，面板刷新可能会慢一会儿。

如果确认已经重启训练、且至少完成一轮 eval 之后仍然为空，再继续查 monitor 链路或日志。

## 11. 一句话判断好坏

- 真正变好：
  `val_completed_rate` 上升，`val_post_speedup_terminated` 下降，`val_post_speedup_steps` 上升，同时 `val_total_score` 不掉
- 表面变好：
  `reward` 上升，但 `val_completed_rate`、`val_total_score` 不动
- 明显变坏：
  `explained_var` 变成接近 `0` 或负数，`value_loss` 持续升高，`val_terminated_rate` 上升

## 12. 早期捡宝箱新增指标

这组指标是 `2026-04-12` 为了观察“前期单怪阶段是否更主动捡宝箱”新增的。

| 指标 | 定义 | 怎么看是好还是坏 |
| --- | --- | --- |
| `{p}_time_to_first_treasure` | 这一局第一次拿到宝箱时的步数；如果整局一个都没拿到，则记为 `max_step + 1` | 越低越好。明显下降说明前期更快拿到第一个箱子；如果长期接近上限，说明还是不会主动开张。 |
| `{p}_pre_speedup_treasures_collected` | 怪物加速前实际拿到的宝箱数量 | 越高越好。这是最直接看“前期会不会捡箱”的指标之一。 |
| `{p}_post_speedup_treasures_collected` | 怪物加速后实际拿到的宝箱数量 | 单独高不一定是好事。若它高但 `pre_*` 低，常说明策略把捡箱拖到了危险阶段。 |
| `{p}_pre_speedup_treasure_rate` | `pre_speedup_treasures_collected / (pre + post)`，如果整局没拿到宝箱则记 `0` | 越接近 `1` 越说明宝箱主要发生在加速前；很适合看“捡箱时机”是不是前移。 |
| `{p}_early_loot_collection_bonus` | 前期单怪窗口里，早拿到宝箱得到的额外奖励累计值 | 越高通常越好，说明 early-loot gate 确实经常触发并成功拿到箱子。 |
| `{p}_early_loot_stall_penalty` | 前期单怪安全窗口里，明明有宝箱目标却长期不靠近时的惩罚累计值 | 越接近 `0` 越好；越负说明还在安全期绕圈、不主动靠近宝箱。 |

建议优先看：

1. `val_time_to_first_treasure`
2. `val_pre_speedup_treasures_collected`
3. `val_pre_speedup_treasure_rate`
4. `val_early_loot_stall_penalty`

如果这四个方向同时变好，基本可以认为“前期更爱捡宝箱”确实学到了。
