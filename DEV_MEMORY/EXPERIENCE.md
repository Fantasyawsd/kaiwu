## 当前记录

### 问题 1：事故后旧 Git 状态不能直接作为恢复依据

- 现象：代码被回退到旧版本，且 Git 文件被删除过；工作区中的 `git status` 同时出现大量旧文件删除与恢复文件新增。
- 影响：不能简单依赖 Git 历史、索引或“当前 diff”来判断事故前的最新工作区状态。
- 根因：事故已经破坏了本地版本控制元信息与工作区一致性。
- 解决方案：
  - 以用户提供的 `项目恢复方案.md`、当前仍存活的关键文件、以及仓库内稳定引用关系为准重建工作区。
  - 用户明确说明会自行删除旧文件和不需要的文件，因此不要因为 Git 里显示 deleted 就盲目恢复旧文件。
- 相关文件：
  - `项目恢复方案.md`
  - `README.md`
  - `KAIWU-WORKFLOW.md`
  - `GLOBAL_DOCS/*`
- 是否已验证：是；恢复后的关键文档链路已闭合

### 问题 2：主文档恢复后，兼容入口仍可能断链

- 现象：`人类开发者AI协作指南.md` 已恢复，但它仍引用 `人类开发者前置知识.md`；同时恢复方案中仍使用 `KAIWU_WORKFLOW.md`、`面向人类开发者的开发文档.md` 这些旧名字。
- 影响：新主文档存在，但旧提示词、旧文档、旧操作记录跳转时会报路径不存在。
- 根因：仓库文档命名已升级，但兼容别名文件未一并恢复。
- 解决方案：
  - 为旧入口补齐轻量兼容别名文件，统一跳转到当前主文档。
  - 用 Markdown 链接巡检确认不存在本地断链。
- 相关文件：
  - `KAIWU_WORKFLOW.md`
  - `人类开发者前置知识.md`
  - `面向人类开发者的开发文档.md`
  - `人类开发者AI协作指南.md`
- 是否已验证：是；Markdown 本地链接巡检结果为无断链

### 问题 3：恢复阶段需要区分“已恢复结构”和“已验证运行”

- 现象：文档和目录结构已恢复，但恢复后未重新执行容器内 smoke test。
- 影响：可以确认“仓库结构和语法状态基本正常”，但还不能把它表述成“恢复后运行验证已完成”。
- 根因：当前环境下先做了静态恢复和 `compileall` 检查，尚未进入完整 Docker 运行链路。
- 解决方案：
  - 在文档中明确区分“历史 smoke 通过”和“恢复后尚未重新运行 smoke test”。
  - 若后续需要运行态确认，再进入容器执行 `python3 train_test.py`。
- 相关文件：
  - `DEV_MEMORY/算法文档.md`
  - `GLOBAL_DOCS/算法文档/agent_ppo_20260409_1733.md`
- 是否已验证：是；当前表述已与实际验证范围一致

### 问题 4：当前机器无法直接完成仓库要求的 smoke test

- 现象：执行 `docker ps` 时无法连接 Docker daemon；执行 `python train_test.py` 时在入口阶段报 `ModuleNotFoundError: No module named 'kaiwudrl'`。
- 影响：当前机器只能完成静态恢复和语法级检查，不能完成仓库约定的容器内训练烟测。
- 根因：
  - Docker Desktop / Docker daemon 当前未运行或不可连接。
  - 仓库依赖 `kaiwudrl` 平台环境，本机裸 Python 环境不具备该依赖。
- 解决方案：
  - 把“恢复后尚未完成运行态验证”明确写进 `DEV_MEMORY`。
  - 等 Docker daemon 恢复后，在正确容器环境中执行 `python3 train_test.py`。
- 相关文件：
  - `train_test.py`
  - `DEV_MEMORY/算法文档.md`
  - `README.md`
  - `AGENTS.md`
- 是否已验证：是；两类错误都已复现并记录
