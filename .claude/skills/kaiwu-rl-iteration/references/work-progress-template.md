# KaiWu 工作进度文档模板

本参考文档用于 `kaiwu-rl-iteration`，约束工作流在仓库根目录维护一份独立的 `工作进度.md`，用于工作记录和规划。

## 1. 为什么需要单独的 `工作进度.md`

在当前 KaiWu 工作流中：

- `实验记录.md` 适合记录实验细节、指标、日志和结论
- `工作进度.md` 适合记录计划、状态、阻塞、阶段进展、决策和下一步

推荐采用双文档联动：
- `工作进度.md`：管过程
- `实验记录.md`：管结果

两份文档必须通过统一的 work item / 轮次 ID 互相引用。

## 2. 文档位置约定

默认约定：

```text
<repo-root>/工作进度.md
```

若仓库中已经存在等价的工作记录文档，则优先复用已有文档，而不是新建重复文档。

## 3. work item / 轮次 ID 规则

每轮都必须分配唯一 ID，例如：

- `iter-001`
- `iter-002`
- `WI-20260408-001`

要求：
- 同一轮在 `工作进度.md` 与 `实验记录.md` 中使用同一个 ID
- commit message、日志说明、分析结果中尽量带上这个 ID

## 4. 建议状态枚举

推荐使用以下状态：

- `PLAN`
- `IN_PROGRESS`
- `RUNNING_SMOKE`
- `RUNNING_FORMAL`
- `REVIEW`
- `BLOCKED`
- `DONE`
- `STOPPED`

状态变化示例：

```text
PLAN -> IN_PROGRESS -> RUNNING_SMOKE -> RUNNING_FORMAL -> REVIEW -> DONE
```

若中途被环境或结果阻塞，可转为：

```text
... -> BLOCKED
```

```md

- 当前状态：PLAN
- 更新时间：2026-04-08 10:30
- 当前分支：
- HEAD commit：
- baseline commit：

- 验收标准：
- 风险等级：低 / 中 / 高

### 当前计划
- [ ] 动作 1
- [ ] 动作 2
- [ ] 动作 3

### 执行记录
- 最小验证：
- 正式实验：
- 日志 / 结果位置：

- 当前结论：
- 决策：继续 / 回退 / 换方向 / 停止

### 阻塞项
- 无 / 具体阻塞说明
```

## 6. 哪些字段必须写

至少要写：

- work item / 轮次 ID
- 当前状态
- 当前分支
- HEAD commit
- baseline commit
- 主假设
- 目标指标
- 当前计划
- 最小验证 / 正式实验位置
- 决策
- 下一步
- 阻塞项
- 对应 `实验记录.md` 条目

## 7. 每个阶段应该更新什么

### Stage 0：Git baseline

更新：
- 当前状态
- 当前分支
- HEAD commit
- baseline commit
- 工作区是否干净

### Stage 1：当前算法快照

更新：
- 当前算法
- 最小验证命令
- 风险或不一致

### Stage 2：候选调研

更新：
- 候选列表
- 选中的主假设
- 暂不做的方向

### Stage 3：实现

更新：
- 当前计划
- 预计修改文件
- 当前状态
- 阻塞项

### Stage 4：实验执行

更新：
- smoke / long run 命令
- 日志位置
- 当前状态

### Stage 5：结果分析

更新：
- 与 baseline 的指标对照
- 当前结论
- 决策
- 下一步

### Stage 6 / 7：提交与停止

更新：
- 结果对应 commit
- 最终状态（`DONE` / `STOPPED`）
- 后续建议

## 8. 常见错误

- 只更新 `实验记录.md`，不更新 `工作进度.md`
- 没有唯一 ID，导致结果无法回溯
- 只写结果，不写计划和下一步
- 只写成功，不写阻塞项
- 状态不更新，导致不知道当前处于哪一阶段
