# CONTRIBUTE

## 1. 目的

本文定义当前仓库的 Git 协作规范、开发前置动作、提交要求与文档同步要求。

本仓库的默认原则是：算法实现、文档维护、版本管理三者必须同步推进。

---

## 2. 分支规范

### 2.1 主分支

- `main`
  - 主体分支。
  - 只接收已经完整实现、完成验证、文档同步完毕的版本。

### 2.2 开发分支

- `feature/<topic>`
  - 功能开发分支。
  - 所有新算法实现、算法优化、治理文档重构、repo-local skill 落地，默认都在该类分支完成。

建议命名示例：

- `feature/agent-ppo-docs`
- `feature/ppo-reward-shaping`
- `feature/repo-skills-bootstrap`

---

## 3. 开发前置动作

在开始任何非微小改动前，至少完成以下动作：

1. 拉取最新代码。
2. 检查当前分支是否适合继续开发。
3. 若在 `main` 上，优先新建 `feature/*` 分支。
4. 阅读：
   - `README.md`
   - `KAIWU-WORKFLOW.md`
   - `GLOBAL_DOCS/算法总表.md`
   - `GLOBAL_DOCS/算法调研.md`
5. 若是算法实现任务，初始化或检查：
   - `DEV_MEMORY/TODO.md`
   - `DEV_MEMORY/CHANGELOG.md`
   - `DEV_MEMORY/EXPERIENCE.md`
   - `DEV_MEMORY/算法文档.md`

---

## 4. 开发规范

### 4.1 实现新算法时的最低要求

要实现新算法或做有意义的算法改动，必须遵循：

1. 先分析当前 baseline。
2. 再查询开发文档与接口约束。
3. 一次只验证一个主方向或主假设。
4. 代码修改后先做最小验证。
5. 文档与代码同步更新。
6. 正式算法实现默认始终在 `agent_ppo/` 中进行；`agent_diy/` 只作为模板参考。

### 4.2 文档同步要求

以下文件是默认需要同步检查的：

- `GLOBAL_DOCS/算法总表.md`
- `GLOBAL_DOCS/算法调研.md`（若本轮是基于外部新调研启动的新方向）
- `GLOBAL_DOCS/算法文档/*`
- `GLOBAL_DOCS/CHANGELOG.md`
- `GLOBAL_DOCS/TODO.md`
- `GLOBAL_DOCS/EXPERIENCE.md`
- `DEV_MEMORY/*`

如果改动只影响当前轮次且尚未稳定，先写 `DEV_MEMORY/*`；如果结论已经稳定且对后续轮次有价值，再归档到 `GLOBAL_DOCS/*`。

补充约定：

- `GLOBAL_DOCS/算法总表.md` 只记录已实现算法。
- `GLOBAL_DOCS/算法调研.md` 记录外部调研得到、但尚未实现的候选方向。

### 4.3 验证规范

当前仓库默认最小验证入口：

```bash
python3 train_test.py
```

如果本轮改动无法通过最小验证，不应做“完整算法实现”收口提交。

---

## 5. commit 规范

### 5.1 小步 commit

适用场景：

- 单个问题修复
- 单个模块重构
- 阶段性文档同步
- 算法实现中的一个可独立验证步骤

小步 commit 信息至少要写清：

- 改动目的
- 改动文件
- 关键接口或配置变化
- 最小验证状态

### 5.2 完整算法 commit

适用场景：

- 一个算法已经完整实现
- 当前阶段算法文档、算法总表、CHANGELOG / TODO / EXPERIENCE 已同步
- 最小验证已通过

完整 commit 信息至少要写清：

- 实现或完成的算法
- 本轮算法完整名（必须显式出现，且应与 `GLOBAL_DOCS/算法总表.md` 中的算法完整名一致）
- 关键超参数设计
- 当前可确认的训练得分或验证状态
- 对应算法文档路径
- 对应的全局 / 任务级文档同步情况

### 5.3 关于训练得分

- 有真实结果时，写真实结果。
- 没有正式训练结果时，明确写“暂无正式训练得分”或“仅完成 smoke 验证”。
- 不允许为了让提交更完整而编造结果。

---

## 6. 完整改动的标准收口

一个“完整改动”至少应满足：

1. 代码已经完成本轮目标。
2. 最小验证通过。
3. `DEV_MEMORY/*` 已同步。
4. 当前算法正式文档已更新。
5. `GLOBAL_DOCS/算法总表.md` 已更新对应条目。
6. 重要变更已写入 `GLOBAL_DOCS/CHANGELOG.md`。
7. 当前 / 后续任务已写入 `GLOBAL_DOCS/TODO.md` 或 `DEV_MEMORY/TODO.md`。
8. 通用经验已写入 `GLOBAL_DOCS/EXPERIENCE.md`，或当前轮次经验已写入 `DEV_MEMORY/EXPERIENCE.md`。

---

## 7. push / merge 规则

推荐流程：

1. 开发完成后先做本地验证。
2. 再做 commit。
3. 用户确认后再 push。
4. 算法完整实现并验证完毕之前，不允许合并到 `main`。

注意：

- `push` 和 `merge` 属于收口动作，不应在需求尚未确认或结果尚未验证时提前执行。
- 若工作区仍有与当前任务无关的改动，应先清理或拆分，再进行版本管理。

---

## 8. 禁止事项

以下行为默认禁止：

- 不读 README / workflow 就直接大改代码。
- 不分析当前 baseline 就直接重构算法主链路。
- 未验证就宣称算法“完整实现”。
- 未同步文档就做“完整改动”提交。
- 完整算法 commit 不写本轮算法完整名。
- 在 `main` 上直接做非微小改动。
- 编造训练得分、实验结论或归档内容。

---

## 9. 推荐实践

- 优先小步提交，避免一次性混入多个方向。
- 实现过程中持续维护 `DEV_MEMORY/*`，不要等上下文快丢了再补写。
- 当某条经验已经证明可复用时，及时归档到 `GLOBAL_DOCS/EXPERIENCE.md`。
- 正式算法文档统一使用 `算法名_时间.md` 命名，并在算法总表中保留最新路径。
